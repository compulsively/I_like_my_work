# Intake Checklist

## Minimum Intake Fields

| Field | Question |
|---|---|
| Goal | What problem should this solve? |
| User | Who uses it or is affected by it? |
| Scenario | When and where does it happen? |
| Scope | What is included this time? |
| Non-scope | What is explicitly not included? |
| Main flow | What happens from start to finish? |
| Output | What should be delivered next? |
| Acceptance | How will people know it is good enough? |

## Difficulty Classification

### Light

Use light scope when:

- one page, one action, or one small change
- no complex states
- no multiple roles
- no sensitive data or high-risk business impact

Minimum output: goal, scope, function, key rule, acceptance.

### Standard

Use standard scope when:

- one normal feature or page flow
- several states or exceptions
- one or two user roles
- fields, permissions, or data source need clarification

Minimum output: goal, roles, scenarios, scope, flow, fields, rules, exceptions, acceptance.

### Complex

Use complex scope when:

- multiple roles, pages, or systems
- cross-step state transitions
- permissions, data, payments, approvals, or launch risks
- multiple teams need the artifact

Minimum output: full flow, state transitions, permissions, field rules, exceptions, metrics, rollout, risks, open decisions.

## Blocking Question Rule

Ask only if the missing answer changes what should be produced.

Good blocking questions:

- Which user role is this for?
- Is the next output a document or a prototype?
- Is this a one-page change or a full workflow?

Non-blocking questions can be marked as assumptions.
