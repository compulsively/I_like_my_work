# Prototype Output Lite

## Purpose

Create prototype-ready output that explains page purpose, user path, layout, components, data, interactions, states, and review focus without overbuilding the process.

## Load Order

1. Read `references/prototype-scale-rules.md`.
2. Read `references/prototype-template.md`.
3. If source material is unclear, use requirement intake first.

## Workflow

1. Identify prototype target:
   - single page
   - multi-page flow
   - page structure only
   - HTML prototype prompt
   - review script
2. Choose size: light, standard, or complex.
3. Define primary user path before component details.
4. Cover important states: normal, empty, loading, error, no permission, disabled, success.
5. Use realistic mock data and mark assumptions.
6. Provide review notes and unresolved questions.

## Output Rules

- Build or describe the actual usable experience, not only a schema.
- Do not invent product facts or system abilities.
- Do not use decorative design as a substitute for task clarity.
- If local visual or component rules are missing, state that the prototype uses a generic baseline.

## Default Output Shape

```markdown
## Prototype Goal

## Scale

## User Path

## Page List

## Page Structure

## Components And Data

## Interactions And States

## Mock Data

## Review Notes

## Open Questions
```
